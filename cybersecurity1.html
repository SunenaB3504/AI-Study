Chapter 1: Web Security Basics

üõ°Ô∏è Why Web Security Matters

Web security is crucial in today's digital landscape. With cyber attacks becoming increasingly sophisticated, understanding fundamental security principles is essential for any web developer or application owner.

Security Impact:
  • Data Protection - Safeguard user data and privacy
  • Trust Building - Maintain user confidence in your application
  • Legal Compliance - Meet regulatory requirements (GDPR, HIPAA, etc.)
  • Business Continuity - Prevent costly security breaches
  • Reputation Management - Protect brand integrity
Security is Everyone's Responsibility:

Security is not just for security professionals. Every developer, from frontend to backend, plays a crucial role in building secure applications. Security should be integrated into every phase of development, not treated as an afterthought.

üîì Common Web Vulnerabilities

Cross-Site Scripting (XSS)

Attackers inject malicious scripts into web pages viewed by other users. Occurs when user input is not properly sanitized before being displayed.

SQL Injection

Attackers manipulate SQL queries by injecting malicious SQL code through user input. Occurs when user input is directly concatenated into SQL statements.

Cross-Site Request Forgery (CSRF)

Attackers trick users into performing unwanted actions on a web application where they're authenticated. Exploits the trust a site has in a user's browser.

Broken Authentication

Flaws in session management, password recovery, or authentication mechanisms that allow attackers to compromise user accounts.

Sensitive Data Exposure

Failure to properly protect sensitive data like passwords, credit cards, or personal information, often due to weak encryption or improper storage.

Security Misconfiguration

Common security settings not properly configured, such as default passwords, unnecessary services enabled, or misconfigured permissions.

üíâ Cross-Site Scripting (XSS) Deep Dive

Types of XSS Attacks

Stored XSS (Persistent)

1. Attacker submits malicious script to server (e.g., comment field)
2. Server stores the script in database without sanitization
3. Server sends script to other users when they view the page
4. Victim's browser executes the malicious script
// Vulnerable PHP code (Stored XSS)
$name = $_POST['name']; // User input
$sql = "INSERT INTO comments (name, message) VALUES ('$name', '$message')";
mysql_query($sql);

// Later, displaying comments
while($row = mysql_fetch_array($result)) {
    echo "<div>" . $row['name'] . ": " . $row['message'] . "</div>";
}

// Attacker input: <script>alert('XSS');</script>

Defending Against XSS

‚úÖ Sanitize user input on both client and server side
‚úÖ Use Content Security Policy (CSP) headers
‚úÖ Escape output when displaying user data
‚úÖ Use frameworks with built-in XSS protection
‚úÖ Validate input on the client side for better UX

XSS Prevention in Practice

// PHP - Using htmlspecialchars for output escaping
$name = htmlspecialchars($_POST['name'], ENT_QUOTES, 'UTF-8');
$message = htmlspecialchars($_POST['message'], ENT_QUOTES, 'UTF-8');

echo "<div>" . $name . ": " . $message . "</div>";

// JavaScript - Using textContent instead of innerHTML
const userComment = document.createElement('div');
userComment.textContent = userInput; // Safe
// userComment.innerHTML = userInput; // Dangerous!

// React - Automatic escaping
function Comment({ author, text }) {
    return (
        <div>
            <h3>{author}</h3> // Automatically escaped
            <p>{text}</p>
        </div>
    );
}

HTTP Security Headers:

Security headers provide additional layers of protection beyond HTTPS. They instruct browsers how to behave when handling your site's content.

üõ°Ô∏è Security Headers

Essential Security Headers

Content Security Policy (CSP)

Prevents XSS attacks by controlling which resources can be loaded and executed.

X-Frame-Options

Prevents clickjacking attacks by controlling whether your site can be embedded in frames.

X-Content-Type-Options

Prevents MIME type sniffing attacks by telling browsers not to guess content types.

Strict-Transport-Security (HSTS)

Forces browsers to use HTTPS connections for all requests to your domain.

Referrer-Policy

Controls how much referrer information is included in requests.

Permissions-Policy

Controls which browser features and APIs can be used on your site.

Implementing Security Headers

// Apache .htaccess
<IfModule mod_headers.c>
    # Content Security Policy
    Header set Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'"

    # Prevent clickjacking
    Header always set X-Frame-Options DENY

    # Prevent MIME sniffing
    Header set X-Content-Type-Options nosniff

    # Force HTTPS
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"

    # Referrer policy
    Header set Referrer-Policy "strict-origin-when-cross-origin"
</IfModule>

// Node.js Express
const express = require('express');
const helmet = require('helmet');

const app = express();

// Use Helmet for automatic security headers
app.use(helmet());

// Custom CSP
app.use(helmet.contentSecurityPolicy({
    directives: {
        defaultSrc: ["'self'"],
        styleSrc: ["'self'", "'unsafe-inline'"],
        scriptSrc: ["'self'"],
        imgSrc: ["'self'", "data:", "https:"],
    },
}));

// Custom headers
app.use((req, res, next) => {
    res.setHeader('X-Frame-Options', 'DENY');
    res.setHeader('X-Content-Type-Options', 'nosniff');
    next();
});

üîç Input Validation and Sanitization

Client-Side vs Server-Side Validation

Defense in Depth:
  • Client-Side - Provides immediate feedback, improves UX
  • Server-Side - Essential security layer, cannot be bypassed
  • Database Level - Additional validation at data storage layer
// Client-side validation (JavaScript)
function validateEmail(email) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    return emailRegex.test(email);
}

function validateInput(input) {
    // Remove potentially dangerous characters
    const sanitized = input.replace(/[<>"']/g, '');

    // Check length
    if (sanitized.length > 100) {
        return false;
    }

    return sanitized;
}

// Server-side validation (Node.js)
const validator = require('validator');

function validateUserInput(req, res, next) {
    const { email, username, password } = req.body;

    // Validate email
    if (!validator.isEmail(email)) {
        return res.status(400).json({ error: 'Invalid email format' });
    }

    // Validate username (alphanumeric, 3-20 chars)
    if (!validator.isAlphanumeric(username) || !validator.isLength(username, { min: 3, max: 20 })) {
        return res.status(400).json({ error: 'Invalid username' });
    }

    // Validate password strength
    if (!validator.isStrongPassword(password, {
        minLength: 8,
        minLowercase: 1,
        minUppercase: 1,
        minNumbers: 1,
        minSymbols: 1
    })) {
        return res.status(400).json({ error: 'Password too weak' });
    }

    next();
}

SQL Injection Prevention

// Vulnerable to SQL injection
const userId = req.params.id;
const query = `SELECT * FROM users WHERE id = ${userId}`;

// Safe with parameterized queries
const query = 'SELECT * FROM users WHERE id = ?';
connection.execute(query, [userId], (err, results) => {
    // Handle results
});

// Using an ORM (even safer)
const user = await User.findByPk(userId);

// Input validation for dynamic queries
function sanitizeForSQL(input) {
    // Remove SQL keywords and special characters
    return input.replace(/['";\\]/g, '');
}

üîê Authentication Security

Password Security Best Practices

// Secure password hashing (Node.js with bcrypt)
const bcrypt = require('bcrypt');

async function hashPassword(password) {
    const saltRounds = 12; // Higher = more secure but slower
    const hashedPassword = await bcrypt.hash(password, saltRounds);
    return hashedPassword;
}

async function verifyPassword(password, hashedPassword) {
    const isValid = await bcrypt.compare(password, hashedPassword);
    return isValid;
}

// Secure session management
const session = require('express-session');

app.use(session({
    secret: process.env.SESSION_SECRET, // Strong, random secret
    resave: false,
    saveUninitialized: false,
    cookie: {
        secure: true, // HTTPS only
        httpOnly: true, // Prevent XSS access
        maxAge: 24 * 60 * 60 * 1000 // 24 hours
    }
}));

// CSRF protection
const csrf = require('csurf');
app.use(csrf({ cookie: true }));

// Add CSRF token to all forms
app.use((req, res, next) => {
    res.locals.csrfToken = req.csrfToken();
    next();
});
