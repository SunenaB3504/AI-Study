üîç What is NoSQL?

NoSQL (Not Only SQL) databases are designed to handle large volumes of unstructured or semi-structured data. Unlike traditional relational databases, NoSQL databases offer:

üöÄ Key Advantages:

  • Scalability: Horizontal scaling across multiple servers
  • Flexibility: Dynamic schemas that can evolve with your application
  • Performance: Optimized for specific data patterns
  • Cost-effective: Can be cheaper for certain workloads

üìä SQL vs NoSQL Comparison

Aspect SQL Databases NoSQL Databases
Data Structure Structured tables with fixed schemas Flexible documents, key-value pairs, graphs
Schema Predefined schema (DDL) Dynamic schema (schema-less)
Scaling Vertical scaling (bigger servers) Horizontal scaling (more servers)
ACID Compliance Full ACID support Varies by database type
Use Cases Complex queries, transactions, structured data Big data, real-time apps, unstructured data
Examples MySQL, PostgreSQL, Oracle MongoDB, Redis, Cassandra, Neo4j

üìÇ Types of NoSQL Databases

üìÑ Document Stores

Store data as documents (typically JSON-like structures). Best for:

  • Content management systems
  • User profiles and preferences
  • Product catalogs
  • Blog posts and articles

üîë Key-Value Stores

Simple key-value pairs for fast lookups. Best for:

  • Caching layers
  • Session storage
  • Real-time analytics
  • Configuration data

üìä Column-Family Stores

Store data in columns rather than rows. Best for:

  • Time-series data
  • IoT sensor data
  • Analytics and reporting
  • Distributed systems

üîó Graph Databases

Store data as nodes and relationships. Best for:

  • Social networks
  • Recommendation engines
  • Fraud detection
  • Knowledge graphs

üçÉ MongoDB: Document Store

MongoDB is the most popular document database, storing data as BSON (Binary JSON) documents.

üìù Basic Operations

üîå Connecting to MongoDB

// Using MongoDB Node.js driver const { MongoClient } = require('mongodb'); async function connectToMongoDB() { const uri = 'mongodb://localhost:27017'; const client = new MongoClient(uri); try { await client.connect(); console.log('Connected to MongoDB'); return client; } catch (error) { console.error('Connection failed:', error); } }

üì• Inserting Documents

// Insert a single document db.users.insertOne({ name: "John Doe", email: "john@example.com", age: 30, hobbies: ["reading", "coding", "gaming"], address: { street: "123 Main St", city: "New York", zipCode: "10001" }, createdAt: new Date() }); // Insert multiple documents db.users.insertMany([ { name: "Jane Smith", email: "jane@example.com", age: 25 }, { name: "Bob Johnson", email: "bob@example.com", age: 35 } ]);

üîç Querying Documents

// Find all users db.users.find(); // Find users older than 25 db.users.find({ age: { $gt: 25 } }); // Find user by email db.users.findOne({ email: "john@example.com" }); // Find users with specific hobby db.users.find({ hobbies: "coding" }); // Complex query with multiple conditions db.users.find({ age: { $gte: 20, $lte: 40 }, "address.city": "New York" });

üîÑ Updating Documents

// Update a single field db.users.updateOne( { email: "john@example.com" }, { $set: { age: 31 } } ); // Add new field to all documents db.users.updateMany( {}, { $set: { lastLogin: new Date() } } ); // Increment a value db.users.updateOne( { email: "john@example.com" }, { $inc: { loginCount: 1 } } ); // Add item to array db.users.updateOne( { email: "john@example.com" }, { $push: { hobbies: "photography" } } );

üóëÔ∏è Deleting Documents

// Delete one document db.users.deleteOne({ email: "john@example.com" }); // Delete multiple documents db.users.deleteMany({ age: { $lt: 18 } }); // Delete all documents db.users.deleteMany({});

üé® Schema Design in NoSQL

üìã Embedding vs Referencing

üîó Embedding (Denormalization)

// Embedded approach - store related data together { _id: ObjectId("..."), title: "Introduction to MongoDB", author: { name: "John Doe", email: "john@example.com", bio: "Database expert" }, content: "MongoDB is a document database...", tags: ["mongodb", "nosql", "database"], comments: [ { user: "Jane Smith", text: "Great article!", createdAt: new Date() } ], createdAt: new Date(), updatedAt: new Date() }

üîó Referencing (Normalization)

// Referenced approach - store data separately // Authors collection { _id: ObjectId("..."), name: "John Doe", email: "john@example.com", bio: "Database expert" } // Posts collection { _id: ObjectId("..."), title: "Introduction to MongoDB", authorId: ObjectId("..."), // Reference to author content: "MongoDB is a document database...", tags: ["mongodb", "nosql", "database"], createdAt: new Date(), updatedAt: new Date() } // Comments collection { _id: ObjectId("..."), postId: ObjectId("..."), // Reference to post userId: ObjectId("..."), // Reference to user text: "Great article!", createdAt: new Date() }

üí° When to Embed vs Reference:

Embed when:
  • Data is frequently accessed together
  • Child documents are bounded in size
  • You need atomic operations on related data
  • One-to-one or one-to-few relationships
Reference when:
  • Data grows unbounded
  • You need to query data independently
  • Many-to-many relationships
  • Data is accessed separately most of the time

üöÄ Aggregation Framework (MongoDB)

MongoDB's aggregation framework allows complex data processing and analysis.

// Basic aggregation pipeline db.orders.aggregate([ // Stage 1: Filter documents { $match: { status: "completed", createdAt: { $gte: new Date("2024-01-01") } } }, // Stage 2: Group by customer and calculate totals { $group: { _id: "$customerId", totalOrders: { $sum: 1 }, totalAmount: { $sum: "$amount" }, averageAmount: { $avg: "$amount" } } }, // Stage 3: Sort by total amount { $sort: { totalAmount: -1 } }, // Stage 4: Limit results { $limit: 10 } ]); // Lookup for joining collections db.posts.aggregate([ { $lookup: { from: "authors", localField: "authorId", foreignField: "_id", as: "author" } }, { $unwind: "$author" }, { $project: { title: 1, content: 1, "author.name": 1, "author.email": 1, createdAt: 1 } } ]);

‚öñÔ∏è Choosing the Right Database

üéØ Decision Factors

üîç Key Considerations:

  • Data Structure: How structured is your data?
  • Query Patterns: What kind of queries do you need?
  • Scalability: How much data and what growth rate?
  • Consistency: How important is ACID compliance?
  • Performance: What are your latency requirements?
  • Development Speed: How fast do you need to iterate?
  • Cost: What's your budget for infrastructure?

üöÄ Best Practices for NoSQL

üí° NoSQL Development Best Practices:

  • Understand your data access patterns before choosing a database
  • Design for your query patterns - optimize for how you read data
  • Consider data consistency requirements - not all NoSQL databases are ACID compliant
  • Plan for horizontal scaling from the beginning
  • Use appropriate indexing to maintain query performance
  • Implement proper error handling and connection pooling
  • Monitor performance metrics and query execution times
  • Backup and disaster recovery are still critical
  • Consider migration strategies if you need to change databases later

üîí Security Considerations

‚ö†Ô∏è Security Best Practices:
  • Enable authentication and authorization
  • Use encrypted connections (SSL/TLS)
  • Implement proper access controls
  • Regular security updates and patches
  • Monitor for unusual activity
  • Secure sensitive data appropriately
