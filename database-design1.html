üìä What is a Relational Database?

A relational database is a type of database that stores and organizes data in tables with relationships between them. Think of it like a spreadsheet, but much more powerful and efficient.

üîë Key Characteristics:

  • Tables: Data is organized in rows and columns
  • Relationships: Tables can be linked through common fields
  • ACID Properties: Atomicity, Consistency, Isolation, Durability
  • SQL: Standard language for querying and managing data

üèóÔ∏è Database Structure

Let's look at a simple example of a university database:

Table Name Purpose Example Columns
students Store student information student_id, first_name, last_name, email, enrollment_date
courses Store course information course_id, course_name, department, credits
enrollments Link students to courses enrollment_id, student_id, course_id, grade, semester

üîß SQL Basics

SQL is the standard language for interacting with relational databases. Let's start with the fundamental commands:

üìù SELECT Statement

The SELECT statement is used to retrieve data from one or more tables:

-- Select all columns from students table SELECT * FROM students; -- Select specific columns SELECT first_name, last_name, email FROM students; -- Select with a condition SELECT * FROM students WHERE enrollment_date > '2023-01-01'; -- Select with ordering SELECT * FROM students ORDER BY last_name ASC;

‚ûï INSERT Statement

The INSERT statement adds new records to a table:

-- Insert a single record INSERT INTO students (first_name, last_name, email, enrollment_date) VALUES ('John', 'Doe', 'john.doe@university.edu', '2024-09-01'); -- Insert multiple records INSERT INTO students (first_name, last_name, email, enrollment_date) VALUES ('Jane', 'Smith', 'jane.smith@university.edu', '2024-09-01'), ('Bob', 'Johnson', 'bob.johnson@university.edu', '2024-09-01');

üîÑ UPDATE Statement

The UPDATE statement modifies existing records:

-- Update a single record UPDATE students SET email = 'john.doe.new@university.edu' WHERE student_id = 1; -- Update multiple records with condition UPDATE students SET enrollment_date = '2024-09-15' WHERE enrollment_date = '2024-09-01';

üóëÔ∏è DELETE Statement

The DELETE statement removes records from a table:

‚ö†Ô∏è Warning: DELETE operations are permanent and cannot be undone. Always use WHERE clauses carefully!
-- Delete a specific record DELETE FROM students WHERE student_id = 1; -- Delete records with condition DELETE FROM students WHERE enrollment_date < '2020-01-01';

üîó JOIN Operations

JOINs allow you to combine data from multiple tables based on related columns:

üîß INNER JOIN

Returns only records that have matching values in both tables:

-- Get student names and their enrolled courses SELECT s.first_name, s.last_name, c.course_name, e.grade FROM students s INNER JOIN enrollments e ON s.student_id = e.student_id INNER JOIN courses c ON e.course_id = c.course_id;

‚¨ÖÔ∏è LEFT JOIN

Returns all records from the left table and matching records from the right table:

-- Get all students and their enrollments (including students with no enrollments) SELECT s.first_name, s.last_name, c.course_name FROM students s LEFT JOIN enrollments e ON s.student_id = e.student_id LEFT JOIN courses c ON e.course_id = c.course_id;

üìä Types of JOINs

JOIN Type Description Use Case
INNER JOIN Only matching records Get students who are enrolled in courses
LEFT JOIN All from left + matching from right Get all students, with course info if enrolled
RIGHT JOIN All from right + matching from left Get all courses, with student info if enrolled
FULL OUTER JOIN All records from both tables Get all students and all courses

üéØ Database Normalization

Normalization is the process of organizing data to minimize redundancy and improve data integrity. There are several normal forms:

1Ô∏è‚É£ First Normal Form (1NF)

Requirements:
  • Each column contains atomic (indivisible) values
  • No repeating groups or arrays
  • Each record is uniquely identifiable

‚ùå Before 1NF:

CREATE TABLE students_bad ( student_id INT PRIMARY KEY, name VARCHAR(100), courses VARCHAR(500) -- "Math,Science,History" - violates 1NF );

‚úÖ After 1NF:

CREATE TABLE students ( student_id INT PRIMARY KEY, first_name VARCHAR(50), last_name VARCHAR(50), email VARCHAR(100) ); CREATE TABLE enrollments ( enrollment_id INT PRIMARY KEY, student_id INT, course_id INT, grade CHAR(1), FOREIGN KEY (student_id) REFERENCES students(student_id), FOREIGN KEY (course_id) REFERENCES courses(course_id) );

2Ô∏è‚É£ Second Normal Form (2NF)

Requirements:
  • Must be in 1NF
  • No partial dependencies (non-key attributes depend on part of the primary key)

3Ô∏è‚É£ Third Normal Form (3NF)

Requirements:
  • Must be in 2NF
  • No transitive dependencies (non-key attributes don't depend on other non-key attributes)

üîë Primary Keys and Foreign Keys

üóùÔ∏è Primary Keys

A primary key uniquely identifies each record in a table:

-- Single column primary key CREATE TABLE users ( user_id INT PRIMARY KEY AUTO_INCREMENT, username VARCHAR(50) UNIQUE, email VARCHAR(100) ); -- Composite primary key CREATE TABLE order_items ( order_id INT, product_id INT, quantity INT, PRIMARY KEY (order_id, product_id) );

üîó Foreign Keys

Foreign keys create relationships between tables:

CREATE TABLE orders ( order_id INT PRIMARY KEY AUTO_INCREMENT, user_id INT, order_date DATE, total_amount DECIMAL(10,2), FOREIGN KEY (user_id) REFERENCES users(user_id) ); CREATE TABLE products ( product_id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100), price DECIMAL(8,2), category_id INT, FOREIGN KEY (category_id) REFERENCES categories(category_id) );

üí° Best Practices for Keys:

  • Use surrogate keys (auto-incrementing integers) for primary keys
  • Name foreign key columns consistently (e.g., table_name_id)
  • Always define foreign key constraints for data integrity
  • Consider using UUIDs for distributed systems

üìä Aggregate Functions and GROUP BY

Aggregate functions perform calculations on groups of rows:

-- Count total students SELECT COUNT(*) FROM students; -- Count students by enrollment year SELECT YEAR(enrollment_date) as year, COUNT(*) as student_count FROM students GROUP BY YEAR(enrollment_date); -- Average grade by course SELECT c.course_name, AVG(CASE WHEN e.grade = 'A' THEN 4.0 WHEN e.grade = 'B' THEN 3.0 WHEN e.grade = 'C' THEN 2.0 WHEN e.grade = 'D' THEN 1.0 ELSE 0.0 END) as avg_gpa FROM courses c LEFT JOIN enrollments e ON c.course_id = e.course_id GROUP BY c.course_id, c.course_name;

üìà Common Aggregate Functions

Function Description Example
COUNT() Count rows COUNT(*) FROM users
SUM() Sum numeric values SUM(price) FROM products
AVG() Average numeric values AVG(salary) FROM employees
MIN() Minimum value MIN(price) FROM products
MAX() Maximum value MAX(salary) FROM employees

üöÄ Best Practices

üí° SQL Development Best Practices:

  • Use meaningful table and column names - Avoid abbreviations
  • Always use WHERE clauses with UPDATE/DELETE - Prevent accidental data loss
  • Use parameterized queries - Prevent SQL injection attacks
  • Add comments to complex queries - Improve maintainability
  • Use transactions for multi-step operations - Ensure data consistency
  • Index frequently queried columns - Improve performance
  • Normalize but don't over-normalize - Balance between integrity and performance

üîí Security Considerations

‚ö†Ô∏è Important Security Practices:
  • Never concatenate user input directly into SQL queries
  • Use prepared statements or parameterized queries
  • Implement proper access controls and permissions
  • Regularly backup your databases
  • Monitor and log database access
