üìö Git Fundamentals

üîç What is Version Control?

Version Control Systems (VCS)

Version control is a system that records changes to files over time, allowing you to:

  • Track Changes: See what changed, when, and by whom
  • Revert Changes: Go back to previous versions
  • Collaborate: Work with others without conflicts
  • Branch & Merge: Work on features independently
  • Backup: Your code is safely stored remotely

üöÄ Why Git?

Git's Advantages:

  • Distributed: Every developer has a full copy of the repository
  • Fast: Operations are local and extremely quick
  • Branching: Lightweight branches encourage experimentation
  • Integrity: Every change is cryptographically signed
  • Free & Open Source: No licensing costs
  • Industry Standard: Used by millions of developers worldwide

üìù Staging & Committing Changes

üîß Add Files to Staging Area

üåø Branching & Merging

üîÄ Branch Management

üîß Create & Switch Branches

# Create and switch to new branch git checkout -b feature/user-auth git switch -c feature/user-auth # Git 2.23+ # Switch to existing branch git checkout main git switch main # Create branch without switching git branch feature/payment-integration # List all branches git branch -a # All branches git branch -r # Remote branches # Delete branch git branch -d feature/completed-feature git branch -D feature/abandoned-feature # Force delete

üîß Merging Branches

# Merge feature branch into main git checkout main git merge feature/user-auth # Merge with custom commit message git merge feature/user-auth -m "Merge user authentication feature" # Abort merge (if conflicts) git merge --abort # Fast-forward vs merge commit git merge --no-ff feature/important # Always create merge commit git merge --ff-only feature/simple # Only allow fast-forward

üîÑ Rebasing

üîß Interactive Rebase

# Rebase current branch on main git checkout feature/my-feature git rebase main # Interactive rebase (last 3 commits) git rebase -i HEAD~3 # During interactive rebase, you can: # pick - use commit # reword - edit message # edit - amend commit # squash - merge with previous # fixup - merge without message # drop - remove commit # Abort rebase git rebase --abort # Continue after resolving conflicts git rebase --continue

üìã Git Best Practices

üí° Essential Git Best Practices:

  • Commit Often: Make small, focused commits with clear messages
  • Write Good Commit Messages: Use imperative mood, be descriptive
  • Use Branches: Keep main branch clean, use feature branches
  • Pull Before Push: Always fetch and merge before pushing
  • Don't Commit Generated Files: Use .gitignore for build artifacts
  • Use Meaningful Branch Names: feature/user-auth, bugfix/login-issue
  • Review Before Merge: Use pull requests for code review
  • Keep Repository Clean: Regular maintenance and cleanup
  • Backup Important Branches: Don't rely solely on local copies
  • Learn from Mistakes: Use git reflog to recover from errors

üìù Commit Message Guidelines

üîß Good Commit Message Format

# Good commit messages follow this format: # Type: Brief description (50 chars max) # Types: # feat: New feature # fix: Bug fix # docs: Documentation changes # style: Code style changes (formatting, etc.) # refactor: Code refactoring # test: Adding or updating tests # chore: Maintenance tasks # Examples: feat: add user authentication with JWT tokens fix: resolve memory leak in image processing docs: update API documentation for v2 endpoints refactor: simplify user validation logic test: add unit tests for payment processing

üö´ Common Mistakes to Avoid

‚ùå Git Anti-Patterns:

  • Committing Large Files: Use Git LFS for large assets
  • Force Pushing: Avoid git push --force on shared branches
  • Committing Secrets: Never commit passwords, API keys, etc.
  • Mixing Unrelated Changes: One commit = one logical change
  • Working Directly on Main: Always use feature branches
  • Ignoring .gitignore: Keep generated files out of version control
  • Not Pulling Regularly: Stay up-to-date with team changes
  • Using git add . Blindly: Review changes before staging
