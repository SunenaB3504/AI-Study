3.1 What Makes an Android Application?

Android applications consist of loosely coupled components, bound using an application manifest that describes each component and how they interact. This modular architecture allows for flexible, maintainable applications that can share components across different apps.

Application Components

Android applications are built from six key component types:

Activities

Your application's presentation layer. Every screen in your application will be an extension of the Activity class. Activities use Views to form graphical user interfaces that display information and respond to user actions. In terms of desktop development, an Activity is equivalent to a Form or Window.

Services

The invisible workers of your application. Service components run invisibly, updating your data sources and visible Activities and triggering Notifications. They're used to perform regular processing that needs to continue even when your application's Activities aren't active or visible.

Content Providers

A shareable data store. Content Providers are used to manage and share application databases. Content Providers are the preferred way of sharing data across application boundaries. This means that you can configure your own Content Providers to permit access from other applications and use Content Providers exposed by others to access their stored data.

Intents

A simple message-passing framework. Using Intents, you can broadcast messages system-wide or to a target Activity or Service, stating your intention to have an action performed. The system will then determine the target(s) that will perform any actions as appropriate.

Broadcast Receivers

Intent broadcast consumers. By creating and registering a Broadcast Receiver, your application can listen for broadcast Intents that match specific filter criteria. Broadcast Receivers will automatically start your application to respond to an incoming Intent, making them ideal for event-driven applications.

Notifications

A user notification framework. Notifications let you signal users without stealing focus or interrupting their current Activities. They're the preferred technique for getting a user's attention from within a Service or Broadcast Receiver.

By decoupling the dependencies between application components, you can share and interchange individual pieces, such as Content Providers or Services, with other applications — both your own and those of third parties.

3.2 Introducing the Application Manifest

Each Android project includes a manifest file, AndroidManifest.xml, stored in the root of the project hierarchy. The manifest lets you define the structure and metadata of your application and its components.

It includes nodes for each of the components (Activities, Services, Content Providers, and Broadcast Receivers) that make up your application and, using Intent Filters and Permissions, determines how they interact with each other and other applications.

It also offers attributes to specify application metadata (like its icon or theme), and additional top-level nodes can be used for security settings and unit tests.

Manifest Structure

The manifest is made up of a root manifest tag with a package attribute set to the project's package. It usually includes an xmlns:android attribute that supplies several system attributes used within the file.

<manifest xmlns:android="http://schemas.android.com/apk/res/android"
          package="com.my_domain.my_app">
          [ ... manifest nodes ... ]
</manifest>

Component Declaration

The manifest supports the following component tags:

  • application - A manifest can contain only one application node. It uses attributes to specify the metadata for your application (including its title, icon, and theme). It also acts as a container that includes the Activity, Service, Content Provider, and Broadcast Receiver tags.
  • activity - An activity tag is required for every Activity displayed by your application, using the android:name attribute to specify the class name.
  • service - As with the activity tag, create a new service tag for each Service class used in your application.
  • provider - Provider tags are used for each of your application's Content Providers.
  • receiver - By adding a receiver tag, you can register a Broadcast Receiver without having to launch your application first.

Security and Permissions

Android includes a comprehensive security model that requires applications to declare the permissions they require. The uses-permission tag declares the permissions your application needs:

<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"/>

Modern Android development emphasizes privacy-first design, with granular permissions and user consent requirements.

3.3 Android Application Life Cycle

Unlike most traditional environments, Android applications have no control over their own life cycles. Instead, application components must listen for changes in the application state and react accordingly, taking particular care to be prepared for untimely termination.

Process Management

As mentioned before, by default, each Android application is run in its own process that's running a separate instance of the Android Runtime (ART). Memory and process management of each application is handled exclusively by the run time.

While uncommon, it's possible to force application components within the same application to run in different processes or to have multiple applications share the same process using the android:process attribute on the affected component nodes within the manifest.

Application Priority and Process States

The order in which processes are killed to reclaim resources is determined by the priority of the hosted applications. An application's priority is equal to its highest-priority component.

Android defines five process states:

  1. Foreground Process - Active processes hosting applications with components currently interacting with the user
  2. Visible Process - Processes hosting "visible" Activities that aren't in the foreground but are still visible
  3. Service Process - Processes hosting Services that have been started
  4. Background Process - Processes hosting Activities that aren't visible and don't have Services
  5. Empty Process - Processes that don't host any active application components

Android aggressively manages its resources, doing whatever it takes to ensure that the device remains responsive. This means that processes (and their hosted applications) will be killed, without warning if necessary, to free resources for higher-priority applications.

3.4 Externalizing Resources

No matter what your development environment, it's always good practice to keep non-code resources like images and string constants external to your code. Android supports the externalization of resources ranging from simple values such as strings and colors to more complex resources like images (drawables), animations, and themes.

By externalizing resources, they become easier to maintain, update, and manage. This also lets you define alternative resource values to support different hardware and internationalization.

Creating Resources

Application resources are stored under the res/ folder of your project hierarchy. In this folder, each of the available resource types can have a subfolder containing its resources.

Simple Values

Supported simple values include strings, colors, dimensions, and string or integer arrays. All simple values are stored within XML files in the res/values folder.

<resources>
  <string name="app_name">My Application</string>
  <color name="primary_color">#FF6200EE</color>
  <dimen name="default_padding">16dp</dimen>
</resources>

Drawables

Drawable resources include bitmaps and vector graphics. They are stored as individual files in the res/drawable folder. Android supports PNG, JPG, GIF, and vector drawable formats.

Layouts

Layout resources let you decouple your presentation layer by designing user-interface layouts in XML rather than constructing them in code. Each layout definition is stored in a separate file in the res/layout folder.

Using Resources

You access resources in code using the static R class. R is a generated class based on your external resources and created by compiling your project.

// Access a string resource
String appName = getString(R.string.app_name);

// Access a color resource
int primaryColor = getColor(R.color.primary_color);

// Access a drawable resource
Drawable icon = getDrawable(R.drawable.app_icon);

3.5 Android Activities

To create user-interface screens for your applications, you extend the Activity class, using Views to provide user interaction. Each Activity represents a screen (similar to the concept of a Form in desktop development) that an application can present to its users.

Creating an Activity

To create a new Activity, you extend the Activity class, defining the user interface and implementing your functionality:

class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
    }
}

The preferred way to design the UI for an Activity is to use a layout resource. The setContentView method accepts either a layout resource ID or a View instance.

Activity Lifecycle

A good understanding of the Activity life cycle is vital to ensure that your application provides a seamless user experience and properly manages its resources.

As activities are created and destroyed, they transition through four possible states:

  • Active/Running - Activity is visible and has focus
  • Paused - Activity is visible but doesn't have focus
  • Stopped - Activity is not visible
  • Destroyed - Activity has been terminated

The Activity class provides several callback methods that you can override to handle these state changes:

  • onCreate() - Called when the activity is first created
  • onStart() - Called when the activity becomes visible
  • onResume() - Called when the activity gains focus
  • onPause() - Called when the activity loses focus
  • onStop() - Called when the activity is no longer visible
  • onDestroy() - Called before the activity is destroyed

3.6 Modern Android Development Practices

As of 2025, Android development has evolved significantly with new tools, frameworks, and best practices that improve code quality, maintainability, and user experience.

Architecture Components

Android Architecture Components provide guidance on app architecture, with libraries for common tasks like lifecycle management, data persistence, and UI data binding:

  • ViewModel - Stores UI-related data that survives configuration changes
  • LiveData - Observable data holder that respects the lifecycle
  • Room - SQLite object mapping library
  • WorkManager - For deferrable, guaranteed background work

Jetpack Compose

Jetpack Compose is Android's modern toolkit for building native UI. It simplifies and accelerates UI development with:

  • Declarative UI programming
  • Less code and powerful tools
  • Intuitive Kotlin APIs
  • Accelerated development

Material Design 3

Material Design 3 provides an updated design system with dynamic color, improved accessibility, and modern components that adapt to different screen sizes and input methods.

Kotlin-First Development

Kotlin is now the preferred language for Android development, offering:

  • Null safety and type safety
  • Concise, expressive syntax
  • Full Java interoperability
  • Modern language features

3.7 Creating Your First Modern Android App

Let's create a simple modern Android application using current best practices. This example will demonstrate:

  • Kotlin programming language
  • Jetpack Compose for UI
  • ViewModel for state management
  • Material Design 3 components

Project Setup

Create a new project in Android Studio:

  1. Open Android Studio
  2. Select "New Project"
  3. Choose "Empty Compose Activity"
  4. Set project name and package
  5. Select minimum SDK (API 21 or higher)

MainActivity.kt

class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            MyAppTheme {
                Surface(
                    modifier = Modifier.fillMaxSize(),
                    color = MaterialTheme.colorScheme.background
                ) {
                    Greeting("Android")
                }
            }
        }
    }
}

@Composable
fun Greeting(name: String, modifier: Modifier = Modifier) {
    Text(
        text = "Hello $name!",
        modifier = modifier
    )
}

@Preview(showBackground = true)
@Composable
fun GreetingPreview() {
    MyAppTheme {
        Greeting("Android")
    }
}

Modern App Architecture

Modern Android apps follow the MVVM (Model-View-ViewModel) pattern:

  • Model - Data and business logic
  • View - UI components (Compose functions)
  • ViewModel - UI state and logic

This separation makes apps more testable, maintainable, and scalable.

3.8 Summary

In this chapter, you learned how to design robust applications using loosely coupled application components bound together using the application manifest. You were introduced to the Android application life cycle, learning how each application's priority is determined by its process state.

You discovered how to externalize resources to create hardware-specific and internationalized versions of your applications. Finally, you were introduced to Activities and their role in the application framework, including how to create new Activities and understand their life cycles.

Key takeaways for modern Android development:

  • Use Kotlin as the primary programming language
  • Follow MVVM architecture with ViewModels and LiveData
  • Leverage Jetpack Compose for modern UI development
  • Implement Material Design 3 for consistent user experience
  • Use Android Architecture Components for robust app structure
  • Externalize resources for internationalization and device adaptation
  • Understand and properly handle the Activity lifecycle
